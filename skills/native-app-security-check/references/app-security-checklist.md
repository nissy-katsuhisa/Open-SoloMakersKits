# iOS / ネイティブアプリ セキュリティチェック

## アプリコード・設定を読めば確認できるもの

- パスワード、アクセストークン、リフレッシュトークン、秘密鍵、APIキーは `UserDefaults` や平文ファイルに保存しない。
- iOS の機密情報は Keychain に保存し、用途に合わせて `kSecAttrAccessibleWhenUnlocked` など適切なアクセシビリティを選ぶ。
- ファイル、DB、キャッシュに残る機密データには Data Protection / 暗号化 / アクセス制御を適用する。
- ログ、クラッシュレポート、分析イベント、スクリーンショット、クリップボードに個人情報やトークンが残らないようにする。
- API通信は HTTPS/TLS を必須にし、ATS 例外は本当に必要なドメインだけに限定する。
- 独自のTLS検証、証明書検証の無効化、開発用プロキシ設定の本番混入を禁止する。
- 認証は OAuth/OIDC/PKCE など標準フローを優先し、端末内だけで認可判断を完結させない。
- Face ID / Touch ID は利便性のための再認証補助として扱い、Keychain のアクセス制御などと組み合わせる。
- 入力値はクライアントとサーバーの両方で検証し、SQL/NoSQL/GraphQL/XML/コマンド注入の入口を減らす。
- カスタムURLスキーム、Universal Links、Deep Links は送信元、パラメータ、リダイレクト先を検証する。
- WebView を使う場合は不要なJavaScriptブリッジ、任意URLロード、混在コンテンツ、ローカルファイルアクセスを制限する。
- 権限、Entitlements、Background Modes、Info.plist の用途説明は最小限にし、使っていない権限を削る。
- 依存ライブラリはSCAや脆弱性DBで確認し、放置されたSDKや署名元不明のバイナリを避ける。
- 暗号化は標準ライブラリを使い、独自暗号、固定IV、ハードコード鍵、弱い乱数を使わない。
- 秘密情報はリポジトリ、ビルド設定、アプリバンドル、CIログ、配布用メタデータに含めない。
- Debugビルド、デバッグメニュー、詳細ログ、テスト用エンドポイント、プロキシ許可設定が本番に入っていないか確認する。
- コード署名、Provisioning Profile、配布証明書、CIの署名鍵管理を確認し、期限切れや過剰権限を避ける。
- Jailbreak / root 検知や改ざん検知は防御の一層として扱い、検知だけに依存した設計にしない。
- SAST、シークレットスキャン、依存関係スキャンをCIに組み込み、毎回の変更で自動チェックする。

## 人間が設計・運用・実機で確認する必要があるもの

- セキュリティ基準を OWASP MASVS / MASTG に対応づけ、対象アプリのリスクに応じて必須チェック項目を決める。
- アプリが扱う機密データを洗い出し、認証情報、個人情報、決済情報、位置情報、ヘルスデータなどを分類する。
- ATS 例外を一度無効化してテストし、HTTP接続や弱いTLS設定が残っていないか確認する。
- 証明書ピンニングを使う場合は、証明書ローテーション、バックアップピン、失敗時の挙動を設計する。
- セッショントークンは短命化し、リフレッシュ、失効、ログアウト、端末紛失時の取り消しを確認する。
- API側でユーザー権限を必ず検証し、IDOR/BOLA、過剰なデータ返却、レート制限不足を確認する。
- サードパーティSDKの権限、通信先、データ収集、既知脆弱性、更新状況を棚卸しする。
- 高リスクアプリでは、難読化、改ざん検知、デバイス完全性チェック、ランタイム保護の必要性を検討する。
- DAST、通信プロキシ、端末上のフォレンジック確認で、通信、保存領域、ログ、キャッシュ、セッション挙動を確認する。
- 主要リリース、認証や決済など高リスク機能の変更、SDK更新後には手動ペンテストまたは重点レビューを行う。
- インシデント時にトークン失効、強制アップデート、機能停止、ユーザー通知、監査ログ確認ができる運用を用意する。
