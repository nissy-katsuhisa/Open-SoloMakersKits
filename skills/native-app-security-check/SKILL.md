# ネイティブアプリ セキュリティチェック

## 概要

このスキルは、iOSまたはネイティブモバイルアプリのセキュリティレビューを行うために使う。具体的なリスクを優先し、確認できる場合は該当ファイルや設定名を示し、実行可能な修正案に落とし込む。

## 収集する入力

- アプリのプラットフォームと構成: iOS、Android、React Native、Flutter、またはネイティブ混在構成。
- アプリのリスク特性: 認証、決済、ヘルスケア、位置情報、企業データ、子ども向け、規制対象データを扱うか。
- ソースファイルと設定: `Info.plist`、Entitlements、ビルド設定、通信コード、保存処理、認証/セッション処理、WebView/Deep Link処理、依存関係定義、CI/リリース設定。
- 周辺サービスの手がかり: Terraform/OpenTofu、CloudFormation、Kubernetes、Dockerfile、Cloud Run、Cloudflare、Supabase、Firebase、Sentry、GitHub Actions、Cloud Build、環境変数名、URL、SDK import、CLIコマンド。
- バックエンド/APIの前提: 認証方式、トークンのライフサイクル、レート制限、権限境界、環境分離、利用クラウドやBaaS/SaaSの責務分界。

## レビュー手順

1. まず `references/service-inventory.md` を読み、コード・設定・IaC・CI/CDから利用サービスの棚卸しを作る。
2. 次に `references/app-security-checklist.md` を読む。
3. アプリのリスクを大まかに分類する:
   - 低: アカウントなし、機密データのローカル保存なし、公開コンテンツ中心。
   - 中: ログイン、ユーザー生成データ、有料機能、個人プロフィールデータを扱う。
   - 高: 決済、ヘルスケア、位置情報履歴、企業データ、規制対象データ、高い悪用リスクを扱う。
4. リスクの高い面から確認する: 秘密情報、ローカル保存、通信、認証/セッション、API権限、Deep Link、WebView、SDK、クラウド/BaaS/SaaS、リリース設定。
5. 推奨事項より先に所見を報告する。重大度は `重大`、`高`、`中`、`低` を使う。
6. 可能な限りファイルパス、設定名、コード参照を含める。
7. 根拠が不足している項目は安全と決めつけず、`要確認` と書く。
8. 最後にリリース判断を短く示す: `リリース停止`、`リリース前に修正`、`監視/バックログ`。

## レビュー観点

- ローカル保存: トークン、認証情報、秘密鍵、個人情報を `UserDefaults`、`SharedPreferences`、平文SQLite、ログ、キャッシュ、保護されていないファイルに保存しない。Keychain、Keystore、暗号化ストア、プラットフォームのデータ保護を優先する。
- 通信: HTTPS/TLSを必須にし、ATSやネットワークセキュリティ例外を最小化し、証明書検証が回避されていないことを確認する。ピンニングはリスクに応じて採用し、ローテーション計画を持つ。
- 認証とセッション: OAuth/OIDC/PKCEなど標準フロー、短命トークン、リフレッシュトークン失効、安全なログアウト、サーバー側認可を優先する。
- APIセキュリティ: オブジェクト単位の認可、レート制限、入力検証、過剰なデータ返却、環境分離を確認する。
- プラットフォーム権限: Entitlements、プライバシー権限、バックグラウンドモード、ペーストボード、スクリーンショット、機密UI状態を最小権限にする。
- Deep LinkとWebView: URLスキーム、Universal Links、リダイレクト、ブリッジ公開、JavaScript設定、ローカルファイルアクセス、信頼できないコンテンツ読み込みを検証する。
- サプライチェーン: SDKの権限、依存関係の古さ、既知脆弱性、バイナリの出所、プライバシー挙動、更新方針を確認する。
- ビルドとリリース: 本番ビルドやCIログから、デバッグフラグ、デバッグメニュー、詳細ログ、開発用エンドポイント、プロキシ許可、秘密情報を取り除く。
- 実行時の耐性: 高リスクアプリではJailbreak/root/改ざん検知、難読化、完全性チェックを検討する。ただし、それだけに依存しない。
- テスト: OWASP MASVS/MASTGに対応づけ、SAST、シークレットスキャン、依存関係スキャン、DAST、プロキシテスト、端末保存領域の確認、主要リリース前の手動レビューを行う。

## 出力形式

レビューでは以下の構成を使う:

- 利用サービスの地図
- 所見
- 要確認
- 推奨修正
- リリース判断

レビューは簡潔で実行可能にする。テスト結果を作らず、確認した内容と未確認の内容を明確に分ける。
