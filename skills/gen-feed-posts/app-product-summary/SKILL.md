# App Product Summary

アプリのコード、文言、設定ファイル、画面構成から、事実ベースの機能整理と推測ベースのポジショニング整理を分けたまま、1ファイルで読める形にまとめる。

# Workflow

## 1. 先に入口を見つける

- まず `references/source-discovery.md` を開く
- アプリ名、プラットフォーム、主要導線が分かるファイルを先に特定する
- まず読む候補:
  - アプリの entry point
  - tabs / routes / navigation 定義
  - onboarding や landing 画面
  - profile、feed、search、group、share のような主要画面
  - localization、copy、manifest、app config
- 読み始める前に、`何のアプリかを説明していそうな画面` と `何ができるかを列挙していそうな画面` を分けて探す

## 2. 事実と推測を分ける

- `主要機能` はコードと文言から確認できる内容だけを書く
- `ターゲット像`、`提供価値`、`ユーザー課題`、`キャッチコピー` は推測を含めてよい
- 実ユーザーの声、KPI、収益性、課金意欲は、根拠がない限り断定しない
- 推測は、画面構成、文言、共有導線、通知設計、課金や継続利用の痕跡から積み上げる

## 3. 主要機能を束ねる

- 機能は画面単位ではなく、ユーザー価値のまとまりで束ねる
- 例えば次の切り口を候補にする:
  - コンテンツ閲覧
  - 作成 / 投稿
  - 検索 / 発見
  - プロフィール / 記録
  - ソーシャル
  - グループ / コミュニティ
  - 共有
  - 課金 / サブスク
  - 管理 / 設定
- アプリ固有の体験が強いなら、その固有概念をそのまま 1 機能の見出しにしてよい

## 4. 出力構成を固定する

- 出力は原則 1 ファイルの markdown にまとめる
- セクション順は固定する:
  1. `プロダクトの要約`
  2. `主要機能`
  3. `予測されるターゲット像`
  4. `提供価値`
  5. `想定されるユーザー課題`
  6. `キャッチコピー`

## 5. 各セクションの書き方

### 1. プロダクトの要約

- 2 から 4 文で、何のアプリかがひと目で分かるように書く
- `誰のどんな行動を支えるか`、`何が残るか`、`何でつながるか`、`どう表現できるか` を優先してまとめる
- カテゴリ名だけで済ませず、体験の芯を書く

### 2. 主要機能

- 機能は意味ごとに束ねる
- 1 機能ごとに、何ができるかを短く列挙する
- 必要なら最後に `根拠:` として主要ファイルを添える

### 3. 予測されるターゲット像

- `コアターゲット` と `行動特性` に分ける
- 年齢や属性を決め打ちせず、行動や価値観から書く
- `なぜその人にこのアプリが刺さるか` まで触れる

### 4. 提供価値

- 機能の説明で終わらせず、ユーザーに起きる変化を書く
- 例えば `効率化`、`安心`、`記録`、`自己理解`、`自己表現`、`つながり`、`収益化`、`継続性`、`達成感` のどれが強いかを見る

### 5. 想定されるユーザー課題

- 各課題には必ずペインのサブタイトルを付ける
- 形式は `### Pain 1. ...` のようにする
- 各課題では、`ユーザーがお金を払ってでも解決したい可能性がある不満` を予測して書く
- 便利さではなく、継続利用や支払い意欲につながる深い困りごとを優先する
- 機能の裏返しではなく、ユーザーの言葉に近い不満として書く

### 6. キャッチコピー

- 3 案前後出す
- 1 案は本命として `推し案` を明記する
- 実装と文言に寄せつつ、広告コピーとして読める短さにする

# Writing Rules

- 日本語で書く
- 断定しすぎず、推測は推測として読める表現にする
- セールス資料のように大げさに盛らない
- 文章は短めにし、1 セクション内で話題を散らさない
- `ユーザー課題` では、なぜその悩みが継続しやすいか、なぜお金を払う余地があるかまで触れる
- 実在しない機能を補完しない

# Output

- 保存先の第一候補は `output/product-analysis/<app-slug>-product-summary.md`
- アプリ名が不明なら、repo 名、bundle id、manifest 名、表示名から推定する
- 既存ファイルを更新する場合は、元の見出し順を崩さずに差し替える
- 最終回答では、保存先 path と、事実ベース / 推測ベースの切り分けを短く伝える
