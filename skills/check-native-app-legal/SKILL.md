# ネイティブアプリ法務チェック

## 流れ

1. `references/app-legal-checklist.md` を読み、基準チェックリストとして使う。3つの確認レーン（コードから確認できるもの、App Store / Google Play / 公開ページから確認できるもの、人間の法務判断が必要なもの）を維持する。
2. 根拠タイトルやURLが必要なときは、`references/source-articles.json` を読む。
3. 修正案、実装手順、App Store / Google Playでの変更、ポリシー文面の変更を提案する前に、`references/remediation-playbook.md` を読む。
4. 利用できるファイル、ストア情報、プライバシーポリシー、利用規約、SDK一覧、権限、分析・広告連携、課金フロー、アカウント削除フロー、ユーザー生成コンテンツ、モデレーション、対象ユーザーからアプリの事実を集める。
5. 確認レーンごとにチェックリストと照合する。情報が足りない項目は、適合扱いにせず `不明` と書く。
6. 公開時のリスクで指摘を優先順位付けする。
   - `P0`: ストア却下の可能性が高い、明確な法務リスク、プライバシーポリシー未整備、削除導線なし、センシティブデータの未開示、子ども向け対応の不足、無許可の規制領域
   - `P1`: 重要な不一致、不明確な同意、不十分な利用規約、SDK開示漏れ、サブスクや広告の説明不足
   - `P2`: 文言整理、証跡整理、運用上の追跡、軽微な修正
7. 指摘を先に出し、その後に前提、最後に同じ3つの確認レーンで分けた短い対応チェックリストを出す。
8. すべての修正提案に、以下を必ず含める。
   - 対象ファイル、管理画面、または運用担当
   - 具体的な変更内容
   - 短い作業手順
   - なぜ必要か
   - `references/remediation-playbook.md` または `references/source-articles.json` にある根拠URL

## 確認ルール

- アプリ挙動、プライバシーポリシー、ストア申告、SDK挙動、権限、スクリーンショットの整合性を確認する。
- 第三者SDKによる収集も、アプリ運営者が開示責任を負う前提で扱う。
- アカウント作成機能がある場合は、アカウント削除導線を確認する。
- 明示的に子ども向けでなくても、子ども・年齢ターゲティングの論点を確認する。
- 有料機能、サブスクリプション、無料トライアル、返金、外部決済リンクが、ストアルールや地域の消費者保護表示と合っているか確認する。
- ユーザー生成コンテンツがある場合は、利用規約、モデレーション、通報、ブロック、権利処理を確認する。
- コード、UI、名称、アイコン、メディア、フォント、地図、API、データセットについて、権利またはライセンスが明確か確認する。
- ヘルスケア、金融、融資、ギャンブル、VPN、位置情報共有、ニュース、子ども向けアプリなどの規制領域を明示する。
- 法務・ポリシー変更を提案するときは、必ず `なぜ必要か` と `根拠ソース` を添える。
- この確認はプロダクトと法務準備の一次確認であり、法的助言ではないことを明記する。未解決の高リスク項目は、弁護士など専門家への確認を勧める。
