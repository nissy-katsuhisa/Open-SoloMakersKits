# 修正対応プレイブック

この資料は、iOS / Android / ネイティブアプリの法務・ストア審査チェックで見つかった不備を、実装・公開ページ・ストア申告・運用の修正手順に落とすために使う。

各提案では、必ず以下を出す。

- `対象`: ファイルパス、App Store Connect / Play Console の画面、サーバー関数、データベース移行、または運用担当
- `修正`: 何を変えるか
- `手順`: 短い実行順
- `なぜ必要か`: 法務・審査・利用者保護上の理由
- `根拠ソース`: 参照した記事名とURL

## プライバシーポリシーの具体化

- `対象`: 公開プライバシーポリシー、アプリ内の法務リンク、App Store / Play Console のプライバシーURL設定
- `修正`: アプリ固有の取得情報、利用目的、保存期間、削除方法、共有先、SDK、問い合わせ先を具体的に書く
- `手順`:
  1. コードとバックエンドのスキーマから、アカウント、プロフィール、生年月日/年齢、端末情報、通知トークン、分析、音楽サービス連携、画像/動画、ユーザー生成コンテンツ、チャット、課金、問い合わせ情報を棚卸しする。
  2. 各データカテゴリについて、利用目的、保存先/処理委託先、保存期間、削除方法、第三者共有の有無を書く。
  3. 分析、認証、データベース、ストレージ、SNS共有、クラッシュ解析、広告、音楽APIなど、ユーザーデータを扱う主要な第三者SDK/外部サービスを明記する。
  4. プライバシーポリシー、App Storeのプライバシー詳細、Google Playのデータセーフティ申告、実際のSDK挙動を照合する。
- `なぜ必要か`: ストア申告とユーザー向け開示は、実際のデータ取得・利用・共有・SDK挙動と一致している必要があるため。
- `根拠ソース`:
  - App Storeのプライバシー詳細: https://developer.apple.com/app-store/app-privacy-details/
  - Google Playユーザーデータポリシー: https://support.google.com/googleplay/android-developer/answer/10144311?hl=en
  - Google Playデータセーフティ セクション入力ガイド: https://support.google.com/googleplay/android-developer/answer/10787469?hl=en
  - スマートフォンのアプリケーション・プライバシーポリシーに関するガイドライン: https://www.mcf.or.jp/temp/sppv/mcf_spappp_guidline.pdf

## ストアのプライバシー申告

- `対象`: App Store Connect のプライバシー詳細、Play Console のデータセーフティ、ストア上のプライバシーURL
- `修正`: ストア申告を、コード、SDK、プライバシーポリシー、バックエンド挙動、公開ページと一致させる
- `手順`:
  1. すべてのデータ取得経路を、ストア申告上のデータ分類に対応付ける。
  2. 自前コードだけでなく、SDKが収集・共有するデータも含める。
  3. 識別子、ユーザーID、分析イベント、診断情報、購入情報、UGC、連絡先/メディア/位置情報、音楽データ、通知トークン、広告/トラッキング関連を確認する。
  4. 広告やトラッキングなど、コード上存在しない申告がある場合は、申告を修正するか、意図した実装として整備する。
- `なぜ必要か`: ストア上の表示と実際の挙動がずれると、審査リスクや誤認表示リスクが高まるため。
- `根拠ソース`:
  - App Storeのプライバシー詳細: https://developer.apple.com/app-store/app-privacy-details/
  - App Store審査ガイドライン: https://developer.apple.com/app-store/review/guidelines/
  - Google Playデータセーフティ セクション入力ガイド: https://support.google.com/googleplay/android-developer/answer/10787469?hl=en

## プライバシーマニフェストとSDK棚卸し

- `対象`: iOS `PrivacyInfo.xcprivacy`、パッケージマニフェスト、依存関係ロックファイル、アーカイブ時のプライバシーレポート
- `修正`: 理由が必要なAPI、トラッキングドメイン、収集データ宣言、第三者SDKのプライバシーマニフェストを確認する
- `手順`:
  1. Swift Package、CocoaPods、バイナリSDK、埋め込みframeworkを棚卸しする。
  2. Xcodeでアーカイブし、Appleのプライバシーレポートを確認する。
  3. 自社コードで使う理由が必要なAPIの宣言漏れを追加・修正する。
  4. プライバシーマニフェストや署名要件に対応していないSDKは更新する。
  5. マニフェスト、プライバシーポリシー、App Storeのプライバシー詳細を照合する。
- `なぜ必要か`: Appleはアプリ本体と第三者SDKに対し、プライバシーマニフェストや理由が必要なAPIの正確な宣言を求めているため。
- `根拠ソース`:
  - アプリまたは第三者SDKへのプライバシーマニフェスト追加: https://developer.apple.com/documentation/bundleresources/adding-a-privacy-manifest-to-your-app-or-third-party-sdk
  - App Storeのプライバシー詳細: https://developer.apple.com/app-store/app-privacy-details/

## ユーザー生成コンテンツ、チャット、通報、ブロック

- `対象`: プロフィール、投稿、コメント、チャット、グループ、画像、モデレーション処理、利用規約
- `修正`: ユーザー生成コンテンツに対して、通報、ブロック、削除/非表示、モデレーション確認、違反対応の導線を追加する
- `手順`:
  1. ユーザー生成コンテンツが出る場所を洗い出す。例: プロフィール項目、ハンドル、自己紹介、投稿、画像、コメント、チャット、グループ、招待、リアクション、リンク。
  2. 不適切・違法・権利侵害の可能性がある場所に、コンテンツ単位/ユーザー単位の通報導線を追加する。
  3. 直接交流がある機能には、ブロックまたはミュートを追加する。
  4. 通報データとして、対象コンテンツID、対象ユーザーID、通報者ID、理由、自由記述、ステータス、日時、対応メモを保存する。
  5. 管理側の運用として、確認キュー、削除基準、エスカレーション、問い合わせ/異議申立て導線を用意する。
  6. 利用規約に、ユーザー生成コンテンツの権利、禁止事項、削除、通報、停止、違反対応を反映する。
- `なぜ必要か`: SNS、チャット、ユーザー生成コンテンツを扱うアプリでは、悪用、権利侵害、違法投稿に対応する安全対策と利用規約上の根拠が必要になるため。
- `根拠ソース`:
  - App Store審査ガイドライン: https://developer.apple.com/app-store/review/guidelines/
  - Google Playアプリ審査準備ガイド: https://support.google.com/googleplay/android-developer/answer/9859455?hl=en
  - App Store／Google Play審査にも対応！アプリ利用規約の作り方を弁護士が徹底解説: https://www.ys-law.jp/IT/column/column-11358/

## 年齢、子ども、未成年対応

- `対象`: サインアップ、年齢ゲート、ストア年齢レーティング、プライバシーポリシー、利用規約、広告/SDK設定
- `修正`: 未成年または13歳未満の利用可否を決め、実装と開示を一致させる
- `手順`:
  1. アプリが子ども向け、混在ユーザー向け、一般/成人向けのどれに近いかを判断する。
  2. 13歳未満を許可しない場合は、アカウント作成前にブロックし、利用条件を説明する。
  3. 子どもの利用を許可する場合は、保護者同意、子ども向けSDK/広告設定、データ最小化、削除権利を整備する。
  4. アプリ内年齢ゲート、App Store / Playの年齢設定、プライバシーポリシー、利用規約を一致させる。
- `なぜ必要か`: 子ども向け、または13歳未満の利用がある場合は、より厳しいプライバシー・同意義務が発生し、ストアの年齢表示とも整合させる必要があるため。
- `根拠ソース`:
  - 児童オンラインプライバシー保護規則（COPPA）: https://www.ftc.gov/legal-library/browse/rules/childrens-online-privacy-protection-rule-coppa
  - App Store審査ガイドライン: https://developer.apple.com/app-store/review/guidelines/
  - Google Playユーザーデータポリシー: https://support.google.com/googleplay/android-developer/answer/10144311?hl=en

## アカウント削除とデータ保持

- `対象`: アプリ内アカウント削除UI、バックエンドのアカウント削除関数、データベーススキーマ、連鎖削除、ストレージ削除、プライバシーポリシー
- `修正`: アカウントに紐づくデータを削除または匿名化し、保持する情報は理由と範囲を説明する
- `手順`:
  1. アプリの削除UIからバックエンドの処理入口までの流れを追う。
  2. アカウントに紐づくテーブル、ストレージ、認証ID、プッシュ通知トークン、分析ID、音楽連携トークン、投稿、コメント/チャット、通報、メディアを洗い出す。
  3. 各カテゴリについて、連鎖削除または明示的な削除/匿名化処理があるか確認する。
  4. 削除範囲をテストで保証する。
  5. 削除される情報と保持される情報をプライバシーポリシーに書く。
- `なぜ必要か`: アカウント作成があるアプリでは、実効性のある削除導線と、削除/保持データの明確な説明が求められるため。
- `根拠ソース`:
  - App Store審査ガイドライン: https://developer.apple.com/app-store/review/guidelines/
  - Google Playユーザーデータポリシー: https://support.google.com/googleplay/android-developer/answer/10144311?hl=en
  - App Storeのプライバシー詳細: https://developer.apple.com/app-store/app-privacy-details/

## 利用規約

- `対象`: 公開利用規約、アプリ内同意文言、サインアップ同意フロー
- `修正`: サービス内容、アカウント規則、UGCの権利、禁止事項、モデレーション、停止、削除、課金、免責、準拠法、問い合わせ先を整える
- `手順`:
  1. アプリ機能と利用規約のカバー範囲を比較する。
  2. ユーザー生成コンテンツを扱うアプリでは、投稿コンテンツの利用許諾と利用者責任を追加する。
  3. 通報、削除、表示制限、アカウント制限、反復違反への対応を追加する。
  4. 課金がある場合は、サブスク/IAP/返金/解約に関する条項を追加する。
  5. アプリ内の同意文言が最新の利用規約とプライバシーポリシーにリンクしているか確認する。
- `なぜ必要か`: 利用規約は、ユーザーの義務、運営者の対応権限、UGCやアカウント措置、課金条件の根拠になるため。
- `根拠ソース`:
  - App Store／Google Play審査にも対応！アプリ利用規約の作り方を弁護士が徹底解説: https://www.ys-law.jp/IT/column/column-11358/
  - App Store審査ガイドライン: https://developer.apple.com/app-store/review/guidelines/

## 課金、サブスクリプション、広告、トラッキング

- `対象`: StoreKit / Google Play Billingコード、ペイウォールUI、商品メタデータ、ストア申告、広告SDK、ATT、プライバシーポリシー
- `修正`: 価格、無料トライアル、自動更新、解約、返金、外部決済リンク、広告、トラッキング、広告識別子を正確に開示する
- `手順`:
  1. StoreKit/Billing、RevenueCat/Adapty、広告SDK、アフィリエイトリンク、IDFA/AAID、ATT、SKAdNetworkを検索する。
  2. 課金がある場合は、ペイウォール文言、商品メタデータ、解約案内、地域別の消費者向け表示を確認する。
  3. 広告やトラッキングがある場合は、同意UI、オプトアウト、プライバシーポリシー、ストア申告、データセーフティを確認する。
  4. 広告やトラッキングがないのにストア上で申告されている場合は、ストア申告を修正する。
- `なぜ必要か`: 課金、広告、トラッキングは、実装と表示が一致しない場合にストア審査・消費者保護・プライバシー上のリスクが大きいため。
- `根拠ソース`:
  - App Store審査ガイドライン: https://developer.apple.com/app-store/review/guidelines/
  - Google Playユーザーデータポリシー: https://support.google.com/googleplay/android-developer/answer/10144311?hl=en
  - Google Playデータセーフティ セクション入力ガイド: https://support.google.com/googleplay/android-developer/answer/10787469?hl=en

## 権利、ライセンス、知的財産

- `対象`: アプリ名、アイコン、フォント、画像、音源、音楽メタデータ/アートワーク、地図、データセット、API、SDKライセンス、生成素材
- `修正`: 各素材・データの権利または利用許諾を確認し、制限事項を記録する
- `手順`:
  1. バンドル素材とリモート素材を棚卸しする。
  2. フォント、アイコン、画像、音楽メタデータ/アートワーク、APIデータ、SDKのライセンスを確認する。
  3. 権利が不明な素材は削除または差し替える。
  4. 必要なクレジットやライセンス表記を追加する。
- `なぜ必要か`: 商標、著作物、フォント、第三者データを権利不明のまま使うと、ストア却下、削除申立て、法的請求につながる可能性があるため。
- `根拠ソース`:
  - モバイルアプリの保護: https://www.wipo.int/publications/en/details.jsp?id=4569
  - App Store審査ガイドライン: https://developer.apple.com/app-store/review/guidelines/
